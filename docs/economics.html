Does it actually pay off?

Faster, cheaper, lighter on context. Which of those is real, and by how much?

Short answer: yes to all three, but only after you account for the prompt you have to add. Here's the arithmetic, with the honest caveats, so you can decide for your own workload.

Short version. Faster is the strong one: generation is decode-bound and sequential, so wall-clock time tracks output-token count almost linearly. ~50% fewer output tokens means roughly ~50% less generation time per file. Cheaper holds after a tiny break-even: output tokens cost more than input, and the one-time prompt is recouped in under one controller (see below). Less context is real but second-order: it's "cheaper" applied to every later turn, since shorter code is re-processed as input each turn.

How an LLM spends your time and money

Two facts drive everything here:

  • Pricing is per token, and output costs more than input, commonly 3 to 5× the input rate. So a token you don't have to emit is worth several input tokens.
  • Latency is dominated by decode, not prefill. The model reads your prompt in parallel (prefill, fast), then emits the answer one token at a time (decode, sequential). Generation time ≈ output tokens × per-token latency. Input length barely moves it, and with prompt caching, re-sent input is nearly free.

Both point the same way: cutting output tokens is what pays. Cutting input characters that don't change token count does nothing.

Crucially this is input, paid once per session and cacheable, not re-emitted per file like output is. That asymmetry (cheap, one-time, cached input vs. expensive, recurring output) is the whole reason the trade works.

In every case you're ahead before finishing the first controller (values < 1). Every controller after that is pure savings: the output tokens you didn't emit, times the price ratio. The more code you generate per session, the better it gets.

Faster, the strongest claim

Because decode is sequential, the file that is half the tokens takes roughly half the time to stream out. The added prompt lands in prefill (parallel, and cached after the first call), so it barely touches latency. Net: shorter time-to-finished-file, and a snappier feel when iterating.

Less context, real but secondary

In a multi-turn session, everything already written stays in the window and is re-processed as input on every subsequent turn. A controller that comes out ~50% smaller leaves more room before you hit the limit (or trigger summarization), and makes each later turn's input cheaper. It's the same mechanism as "cheaper," applied forward in time.

Where the claim breaks down

  • The tokenizer here is a proxy. o200k_base is GPT-4o's, and Claude's differs. The ratios hold qualitatively, the exact numbers won't.
  • Unusual names are out-of-distribution. The model has seen SaveChangesAsync billions of times and save() almost never. With the prompt in context it's reliable, but expect the occasional slip, and a correction turn can cost more than the file saved. Adherence matters, so measure it on your stack.
  • Reasoning tokens are unaffected. On a thinking model, the compact style shrinks the answer, not the hidden reasoning it spends working out the logic.
  • Tiny one-off edits don't amortize. If you generate one three-line snippet and stop, the prompt overhead can exceed the savings. The trade favors output-heavy, multi-file, multi-turn work.
  • Character-only shortcuts give none of this. .Where( → .w( is the same token count, so zero cost or speed benefit. We keep a few for consistency and label them 0 in the reference.
Verdict. For an AI assistant generating ASP.NET Core / EF Core code across a session, all three benefits are real: faster (decode-bound, the cleanest win), cheaper (after a sub-one-controller break-even), and lighter on context (compounding over turns). It is not a win for one-shot tiny edits, and it never changes runtime behavior, only how many tokens the code costs to write.
